目前主要存在的问题

1、没有适配 exectuor接口

如果异步框架需要引入, 衔接起来并不友好

比如 RxJava

   Flowable.just(1, 2)
                .observeOn(Schedulers.from(mExecutor))
                .forEach(new Consumer<Integer>() {
                    @Override
                    public void accept(final Integer integer) throws Exception {
                        
                    }
                });

再比如 CompleteFuture

  CompletableFuture.runAsync(new Runnable() {
            @Override
            public void run() {

            }
        }, mExecutor).thenApply(new Function<Void, Object>() {
            @Override
            public Object apply(final Void aVoid) {
                return null;
            }
        });

我们支持executor以后可以和这些api甚至第三方api无缝对接 同时也支持业务上的一些报警能力

2、提供存储和执行的数据结构性能问题

目前采用的是 LinkedBlockingQueue jdk里面采用的策略是 通过 两把 ReetrantLock来控制 list的数据添加和删除

 /** Lock held by take, poll, etc */
    private final ReentrantLock takeLock = new ReentrantLock();

    /** Wait queue for waiting takes */
    private final Condition notEmpty = takeLock.newCondition();

    /** Lock held by put, offer, etc */
    private final ReentrantLock putLock = new ReentrantLock();

具体细节不展开了, 线程比较多的时候一个问题是锁粒度大, 对于io密集型应用这个时候采用乐观锁可能可以获得更好的性能, java里面 Thread.sleep() LockSuppar.park() Object.wait() 这些操作最终都会追溯到 hotspot vm里面 pthread_cond_wait这个函数, 总之是大概率让出cpu执行时间片, 切换cpu三级缓存行以及相应寄存器变量从而带来较大开销

具体不多说了, 面试题都能搜到一堆

2、non-blockingqueue

MpscArrayQueue jctools和netty里均有实现 多生产者单消费者 因为单线程消费所以消费者index可以完全无锁化

abstract class MpscArrayQueueL3Pad<E> extends MpscArrayQueueConsumerIndexField<E>
{
    long p01, p02, p03, p04, p05, p06, p07;
    long p10, p11, p12, p13, p14, p15, p16, p17;

维护消费者和生产者下标，对于多线程情况采用缓存行填充避免伪共享问题 减少cpu cache-line额外开销, 这个方案属于对性能比较高的一种解决方案

executor的策略

目前我个人倾向于提供多种方案

1、老的jdk提供的线程池, 本质上是个executor

2、多线程对应一个queue, queue可以选择 blockingqueue或者其他的高性能解决方案

3、EventLoop, 每个线程绑定一个queue, queue的选择方案如上, 有锁开销的地方在于runnable分配，这个地方会有一个cas锁，读写比例均等 atomicInteger就够了, queue的写入需要支持线程安全, 通过cas和缓存行填充提高读写性能, 空间换时间

额外工作

1、runnable执行时间统计

2、额外的一个thread支持检测超时任务

3、报警系列代码，这部分应当直接copy目前的代码即可

轮训(Polling)

轮训流程如下

sequenceDiagram
    participant client
    participant server
    client->>server: request start
    loop check
        server->>client: not finished
        client->>server: check status
    end

    server->>client: response finished

服务端代码示例

客户端不断请求服务端, 服务端尽量马上回应并显示状态

长轮训(long polling)

sequenceDiagram
    participant client
    participant server

    client->>server: request

    loop check
        server->>server: wait for task finished
    end

    server->>client: finished task

其实和轮训一样，但是这个时候有可能存在耗时任务或者数据消息尚未存在,　我们hold住这个链接并不释放，等待服务端返回

server端核心代码 这里主要依赖 netty
首先保存这个链接

其次找个适当时机处理这个链接返回数据 可以用网页来模拟

sse(server-sent event)

大致流程如下 客户端发出请求以后, 服务端不断实时地返回数据, 比如更新进度条

sequenceDiagram
    participant client
    participant server

    client->>server: request

    loop check
        server->>client: task status update
    end

sse其实就是一次http请求以后　服务端不断通过该次tcp连接不断向client端发送数据是单向数据流, 在spring里面, 我们已经有了SseEmitter, 不需要做很多额外工作就可以和前端交互

示例代码

websocket(websocket-server)

sequenceDiagram
    participant client
    participant server

    client->>server: http-request
    server->>client: http-response handle-shake

    loop check
        client->>server: sent websocketFrame 发送消息
        server->>client: receive websocketFrame 接受消息
    end


    loop check
        client->>server: ping websocketFrame 心跳发送
        server->>client: pong websocketFrame 回应心跳
    end

    client->>server: close

websocket第一次请求是一次get请求 然后进行协议升级upgrade为websocket
该协议为全双工协议，提供纯文本和二进制数据传输, 提供心跳检测的能力 PingWebsocketFrame和PongWebSocketFrame

垂直拆分分库分表实践

Posted on 2020-01-19

数据数据迁移其实主要就是垂直拆分和分库分表 垂直拆分和分库分表过程中主要数据库的操作就是双写和查询 我们会有开关来控制状态的转换，公司业务里orm主要是用mybatis

本文主要目的是为了减少代码的侵入性和迁移过程中数据库读写代码的可复用性，实际项目里单个表涉及的查询多达几十个，并且涉及到几十个文件的修改，为了减少迁移过程中对业务代码的修改，我文章下面会给一个样例(毕竟不能带上公司业务代码)。我目前在酷家乐工作中有遇到以下两个之前处理起来代码比较繁琐的地方

1、一个是大量的业务表从老的数据进行迁移(这里可能容易遇到自增主键切换写顺序一致性问题)，

2、还有一个问题就是部分大表的扩容(实际上相当于垂直拆分 然后分库) 本质上都是一回事情

进行迁移的话会需要采用dts或者数据库日志binlog同步存量数据的过程，这里根据自己公司的技术栈来选择 存量数据写入数据库的时候带上主键

写增量数据

1、通常做法是进行双写 写入老表的时候同时写入新表，如果需要的话可以加上手动事务管理，毕竟是跨库，不过实际应用场景中写入失败的情况很少，根据实际情况来决定。
2、双写逻辑会麻烦点的地方就是插入顺序切换时候的自增主键一致性问题。

自增主键插入的问题

(1) 老表如果是单表自增的，新表是单表自增的话

找流量低的时候切换读写顺序，如果业务需要高度一致性，加分布式锁，需要额外的开关来决定是否走有锁的逻辑，顺序切换以后，关闭那个控制是否走锁的逻辑的开关(如果条件允许 的话，各个服务器之间其实通过本地rocksDb写磁盘 实现一个分布式一致性算法比如于raft 也是可以的 实际上各个服务自己的集群就是一个小型的raft集群了) 其实这里也可以方法 (2)去解决，就是需要个过渡表来管理主键自增

(2) 老表如果是单表自增的，新表是分表的话

老表自增id 增加一个大区间比如原来是 id = 10^7 我们直接增加到 id = 2 * 10 ^ 7 新表设置自增id为 10^7 + delta( delta > 0 && delta < 10 ^ 4) 这个范围自己控制一下就好, 切换插入顺序以后主键不会冲突，也不会阻塞依赖业务方修改sql为rpc 迁移完成以后 新表主键改成 3 * 10 ^ 7, 这里只是大致数量, 实际区间大小由业务来决定
